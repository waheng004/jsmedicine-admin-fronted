# AGENTS.md

## 项目概览

`中医在线` 管理端前端项目，面向后台管理员提供登录、账户管理、系统管理、首页管理、课程管理、图书管理、资讯管理、播客管理、专题管理、专家管理、题库管理、统计管理、直播管理、答疑管理、反馈管理等完整后台操作能力。

- 面向管理端提供基于浏览器的后台管理界面。
- 当前项目使用 Vue 3 + Vite 构建，先完成登录闭环，再按业务模块逐步扩展。
- 开发过程覆盖 MVP、联调、验收、上线前质量保障，不只停留在页面能打开。
- 优先复用成熟方案和现成依赖，必要时允许在 `package.json` 中新增成熟库，避免重复造轮子。
- 不要过早引入复杂前端架构，保持单体前端、模块清晰、按功能演进。

## 运行环境要求

### 硬件

- 开发机 CPU 4 核及以上。
- 内存 8G 及以上。

### 软件

| 类型 | 名称 | 版本 | 用途 |
|---|---|---:|---|
| Node.js | Node.js | 18+ | 前端构建与开发服务 |
| 包管理器 | npm | 9+ | 依赖安装与脚本执行 |
| 浏览器 | Chrome | 72.0.3626+ | 管理端访问与联调 |
| 后端服务 | 中医在线后端 | 以联调环境为准 | 提供 RESTful API |
| 应用发布 | Nginx | 1.14.0+ | 静态资源发布与反向代理 |

## 技术栈

| 组件 | 说明 |
|---|---|
| Vue 3 | 主框架，优先使用 Composition API |
| Vite | 开发服务器与构建工具 |
| Vue Router | 路由、登录态拦截、页面跳转 |
| Fetch / Axios | HTTP 请求；当前可使用 Fetch，复杂拦截器需求出现后再引入 Axios |
| Cookie | 当前用于记住账号密码和保存登录态 |
| CSS | 当前使用全局 CSS；模块增多后可按组件拆分样式 |
| ESLint / Prettier | 后续引入后作为代码风格与格式化基线 |
| Pinia | 后续出现跨页面共享状态时优先引入，不提前滥用 |
| Element Plus / Ant Design Vue | 后续管理端表格、表单、弹窗大量出现时可选择其一引入，避免混用多个 UI 框架 |

## 开发计划与进度追踪

- 项目根目录的 `Todos.md` 是后续开发阶段、任务优先级和验收状态的计划来源。
- 每次开始新功能或新阶段前，必须先阅读 `Todos.md`，确认当前应推进的阶段和任务，不要跳过前置依赖。
- 如果用户没有指定具体任务，应优先选择 `Todos.md` 中当前 `进行中` 阶段的未完成任务；当前阶段完成后再进入下一阶段。
- 开发过程中如果发现任务拆分不合理、存在阻塞或需要调整顺序，必须同步更新 `Todos.md`，并在回复中说明调整原因。
- 每完成一个任务，必须更新 `Todos.md` 中对应复选框、阶段状态和必要的验收记录。
- 每完成一个阶段，必须确认该阶段验收标准全部满足；未满足时保留阶段为 `进行中` 或标记阻塞，不要提前标记完成。
- `Todos.md` 只记录项目推进计划和验收状态，不替代接口文档、组件文档或代码注释。

## 架构原则

- 页面组件只负责页面展示、用户交互和调用业务 API。
- API 模块只负责请求路径、请求参数、响应处理，不写页面逻辑。
- 通用工具放入 `src/utils/`，避免在页面中复制粘贴。
- 跨页面共享状态出现后放入 `src/stores/`，优先使用 Pinia。
- 路由配置集中管理，登录态和权限拦截统一在 Router 或权限模块中处理。
- DTO / ViewModel 与后端返回结构要保持清晰边界，不直接把复杂后端结构散落到模板里。
- 模块按业务域拆分，避免所有页面堆在一个目录。
- 不要增加无必要的中间层和抽象层。
- 能直接复用现成依赖解决的问题，不手写低价值基础设施代码。
- 简单表格、分页、表单、弹窗后续优先复用成熟 UI 组件库；不要为每个模块重复写平行组件。

## 功能范围

前端实现需覆盖管理端使用手册中的核心业务能力：

- 用户登录、记住账号密码、自动登录、退出登录、权限识别。
- 账户管理：用户管理、学员管理。
- 系统管理：管理员管理、角色管理、权限分配。
- 首页管理：分类管理、首页内容配置。
- 课程管理：课程列表、审核、考卷、视频配置。
- 图书管理：图书列表、分类、审核、章节、考卷配置。
- 资讯管理：帖子新增、富文本编辑、删除、审核。
- 播客管理：音频管理、配置、审核。
- 专题管理：课程、书本、学员、资讯、博客、题库等关联配置。
- 专家管理：专家信息、履历、分类管理。
- 题库管理：题目新增、修改、删除。
- 统计管理：学时、学员、地区、成绩统计。
- 直播管理：直播新增、修改、删除、审核。
- 答疑管理：问题查看、回复查看、删除。
- 反馈管理：反馈查看、删除。

## API 规范

- 前端请求统一走 `src/api/` 下的模块文件。
- RESTful 接口路径以 `/api/v1/...` 为准。
- 分页参数统一使用 `page`、`size`、`sort`。
- 认证统一携带 `Authorization: Bearer <token>`。
- 统一处理后端响应结构，不在每个页面重复判断 `success`、`code`、`message`。
- API 模块必须明确请求参数和返回字段用途，复杂接口需要在旁边补充简短说明。
- 接口变更时先确认后端 Swagger / API 文档，再改页面。
- 开发环境通过 `vite.config.js` 代理 `/api` 到联调后端，避免浏览器跨域影响开发。
- 不要在页面组件里硬编码测试环境域名；环境差异通过 Vite 环境变量或代理配置处理。

## 路由与权限

- 登录页路径统一为 `/login`。
- 登录成功后进入管理端首页；当前临时成功页为 `/success`。
- 需要登录的页面必须配置路由守卫，未登录跳转 `/login`。
- 已登录访问 `/login` 时应跳转到管理端首页或当前可访问的默认页。
- 权限按钮、菜单、路由访问必须以后端返回的角色和权限为准。
- 菜单可见性和路由权限不要只依赖前端隐藏，关键接口仍以后端权限校验为准。
- 退出登录必须清除登录态；如果保留记住账号密码，不应导致退出后立刻自动登录。

## 状态与存储

- Token、记住账号密码当前使用 Cookie 保存。
- 记住账号密码是当前需求，但明文保存密码存在安全风险；上线前必须与后端和产品确认是否改为只记住账号或使用更安全方案。
- 登录态、管理员信息、权限列表应集中读写，不要在多个页面各自维护。
- 临时 UI 状态优先放组件内部，不上升到全局 Store。
- 跨页面状态必须有明确业务原因后再进入 Store。
- 不要把大对象、敏感信息或后端原始响应无控制地塞进 Cookie / localStorage。

## UI 与交互规范

- 管理端页面应保持清晰、克制、适合高频操作，不做营销式首页。
- 表格、筛选、分页、批量操作、审核操作、弹窗确认是管理端核心交互，后续必须形成统一样式。
- 表单必须有校验、加载中、成功、失败、禁用等状态。
- 删除、审核拒绝、批量操作等高风险动作必须有确认。
- 所有异步操作必须有清晰反馈，不让用户不知道是否执行成功。
- 页面文字使用中文，代码注释使用英文。
- 图标优先使用成熟图标库或统一内联 SVG，不要使用粗糙的临时 CSS 图形。
- 响应式至少保证常见桌面分辨率可用；登录页需兼容移动端窄屏。
- 文本不能溢出按钮、表格、卡片或弹窗；长内容必须有换行、省略或弹窗详情策略。

## 样式规范

- 当前全局样式放在 `src/styles.css`。
- 组件增多后，页面私有样式优先放在对应 `.vue` 文件的 `<style scoped>`，通用样式再抽到公共 CSS。
- 不要在多个页面复制同一套按钮、表格、表单样式；需要时提取公共组件或公共 class。
- 不使用大面积单一色相堆叠，管理端应保持层级清楚、对比足够。
- 颜色、间距、字号后续应逐步收敛为设计变量，避免每个页面自行发挥。

## 接口联调与文档

- 联调前先确认后端 Swagger JSON 或接口文档中的路径、方法、请求体、响应体、错误码。
- 每完成一个页面模块，应记录已对接的接口、剩余接口和阻塞项。
- 如果后端接口缺失、字段不一致或错误码不稳定，应记录到 `Todos.md` 或对应联调记录，不要在前端写大量临时兼容逻辑掩盖问题。
- 前端不负责导出 Swagger JSON，但必须以最新后端 `api/api.json` 或 Swagger UI 为接口契约来源。

## 测试与质量

- 每次修改后至少执行 `npm run build`。
- 登录、权限、审核、配置、统计、导出等关键流程后续必须补测试。
- 引入测试框架后，优先覆盖：
  - API 封装的成功与失败响应处理。
  - 路由守卫的登录态跳转。
  - 表单校验和提交状态。
  - 权限菜单和按钮可见性。
  - 核心业务页面的列表、筛选、分页、审核动作。
- 重要页面改动后必须用浏览器实际打开验证，不只看构建是否通过。
- 生产前必须检查：接口契约、异常提示、登录态过期、权限、空状态、加载状态、移动端基础可用性。

## 日志与错误处理

- 前端错误提示必须面向用户可理解，不直接暴露后端堆栈或技术细节。
- 控制台日志只用于开发调试，上线前清理无意义日志。
- API 错误应保留后端 `message`，但需要兜底默认提示。
- 登录过期或 401 响应应统一清理登录态并跳转登录页。
- 403 权限不足应展示明确提示或无权限页面，不要静默失败。

## 安全

- 不在代码中硬编码生产密钥、管理员密码、私有 Token。
- 不把敏感配置提交到仓库。
- 不信任前端权限控制，关键操作必须依赖后端鉴权。
- 富文本、用户输入、文件 URL 展示时必须考虑 XSS 风险。
- 文件上传后续必须限制类型、大小，并以后端返回结果为准。
- Cookie 中保存密码是当前临时需求；上线前应推动改为更安全的记住登录方案。

## 重要事项

- 始终用中文回复。
- 后续开发必须按 `Todos.md` 的阶段安排推进，并在任务完成、阶段调整或发现阻塞时同步更新该文件。
- 优先使用成熟依赖、官方推荐方案和现成组件库，必要时直接在 `package.json` 中补充，不要手动造轮子。
- 先确认接口契约，再实现页面、状态和数据联动。
- 后续新增功能必须优先复用现有 API 封装、路由守卫、登录态工具、统一响应处理和公共组件；不要重复创建平行基础设施。
- 不要为了短期交付牺牲可维护性。
- 面向上线质量开发：可观测性、异常处理、权限、回滚、兼容性都要考虑。
